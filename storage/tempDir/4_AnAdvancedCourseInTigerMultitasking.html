General comments on software development

Avoid faults before they arise. Here are some useful tips:

  • Choose USER_VAR-STRICT (always)
  • Always initialize string: e.g. A$=’’’’, B$=FILL$(“-“, 4k), etc.
  • Initialize variables
  • Use as much information as possible in abort conditions: test at the end of loops, rou tines, etc:
    Wherever possible use:
    IF A >= B THEN ...
    Instead of:
    IF A=B THEN
  • No “trick” programming, keep the use of variables as clear as possible:
    Choose easily understandable names for variables (even long names by no means enlarge the resulting program code)
    Only use variables for one purpose wherever possible. Thus, do not use “N” in the same routine as a control variable in a loop, later as a temperature value and then as a flag for another routine, The supposed advantage of saving some variable space costs a lot of debug time,reduces the clarity and in developing projects finally leads to incalculable risks.

Documentation

Numerous investigations into the freedom from errors and defectiveness of software and the productivity of software developers have been carried out. And the results are almost always the same. The productivity of a software developer is a disappointing 2 to 5 code lines per hour during the complete development time. It is less in larger projects than in shorter projects.

Such investigations also clearly show that errors which are not detected become more expensive the earlier they occur in the course of the project.

Thus, if we look at the plain facts one can hardly make the pressure of deadlines responsible for faulty documentation. With an average time needed of 12 to 30 minutes for one line of a program (!), there is enough time to explain the intention behind a program line, to clearly point out the data structure or select plausible names for variables.

Good, easily understandable documentation is an essential prerequisite for long-term stable projects as well as good update service and later extensions.

Using symbolic values

Keep numerical values and other constants out of the program body as far as possible and define them in the heading of a program. This facilitates an understanding and makes a decisive contribution to better updates.

 

Bad:

Area1 = 51


Slightly better:

AREA1 = 3 * 17

 

Much better:

LENGTH1 = 17 ' 17 meters, length of area-1
WIDTH1 = 3 ' 3 meters, width of area-1
AREA1 = LENGTH1 * WIDTH1 ' square meters in unloading area "1"

A constant such as "17" is difficult to locate in a program if, as in the first case, a derived result (area = 51) has already been used in the program code.

It is slightly better if at least the components of the new value AREA 1 can be recognized as the product of 3 and 17, as done in the second example.

It may also be difficult to immediately recognize the correct meaning of numbers in the program body. Imagine if the number "3" appears 28 times in a program and represents 6 different facts. Have fun looking for the error.

And finally, just imagine what would happen if you want to change certain numerical values in a program. This is easy if all constants are defined and documented in the heading. The search for the numerical value – either in its pure form or already integrated in other numbers – in numerous program files this becomes an extremely time-consuming procedure which is prone to errors.

Elementary values are defined, derived values are calculated

Example: Volume of a cuboid.

B = 12 			' width in mm
H = 50		 	' height in mm
L = 100 		' length
V = 60000 		' volume in mm**3

Bad procedure. One of the 4 values is superfluous and reduces the orthogonality. Imagine the effect if you would like to work with a different volume work elsewhere in the program flow and the assignment: V = 80000 is used. The chaos becomes perfect: B * H * L <> V.

Thus:

B = 16 		' new width (mm)
V = B*H*L 	' calculate new volume (mm**3)

 

Carry out counterchecks, discover errors, weaknesses or limits:

Once a project has almost been completed and already runs "quite well" you should induce some "stress situations" to check the project for stability, error tolerance and concealed bugs.

Such tests can reveal concealed weaknesses or rare errors before the project is installed for the first time or goes into series.

Naturally, you should always pay attention to all applicable safety regulations during such tests. Particularly when working with electric voltages, high energies, forces, dangerous substances and the like in the project. The testing environment must reliably exclude any damage to humans, material and the environment.

How does the project behave in such cases?

  • Preset all variables with difficult/implausible values
  • Induce implausible input conditions or input data
  • Test implausible, contradictory operations
  • Induce specific worst-case situations
  • Test the project at its limits
  • "Real world" tests, installation under realistic conditions.
  • "Idiot test" – have product used by people with no or little advance information.

System definitions

There are a few standard include files which contain definitions. The files: DEFINE_A.INC, DEFINE_I.INC, UFUNCn.INC and KEYB_PP.INC should be included in every larger application. Always use the standard include files in their unmodified, original form and write your own definitions, routines or code pieces in your own include files. This ensures that your own codes will never be overwritten by an update.

DEFINE_A.INC contains definitions that are outside a task.

DEFINE_I.INC contains instructions that must be within a task.

UFUNCn.INC contains definitions for device drivers. New versions of this file increment the ‘n’ value.

KEYB_PP.INC contains both definitions and instructions as well as the subroutine INIT_ KEYB (Device No..) to initialize the LCD1 driver

An advanced course in Tiger MultitaskingVariable definition

Variable definition

Tiger Multitasking allows an explicit or automatic declaration of variables. It is possible to introduce variables without declaring them in smaller programs. However, we recommend an explicit definition of all variables for programs with hundreds or thousands of lines of code. Tiger Multitasking has the compiler directive USER_VAR_STRICT to help the programmer in such cases. If this is used, the compiler reports undefined variables or constants as an error.

The advantage of an explicit variable definition is the clarity achieved in terms of variable type and scope of application. The storage location used can also be better controlled. For example, strings are to be declared explicitly with the maximum length required, byte and word variables are defined as such.

Declaring a variable’s type alone does not assign a value to it. This is important if variables should keep their values within systems using RAM battery backup. After a system power down global and task-local variables still have the same values as before the power down. On the other hand they must be assigned on the first start.

Local variables in subroutines can never be retained by a battery backup! If no protected RAM is provided the variables must be initialized accordingly.

Names in Tiger Multitasking programs can be up to 32 characters long. This can be comprised of alphanumeric and underscore characters. Either lower or upper case letters can be used. The following methods have proven their worth when naming constants and variables:

 

  • the bigger the scope of a variable’s application, the more striking its name
  • do not use names of global variables for local variables
  • the variable type is identified by the 1st letter of its name, e.g.:
    bReceive = BYTE
    wCount = WORD
    lAbsolut_X = LONG
    rAngle_Speed4 = REAL
  • Long names should be optically edited: “HighSchoolEntrance” or “High_School_Entrance” instead of HIGHSCHOOLENTRANCE

Correct use of variable types

One group common variable types in Tiger Multitasking are the integer variables: BYTE, WORD and LONG. When assigning one type to a variable of a different type Tiger Multitasking automatically ensures a corresponding type conversion.

Remember:
BYTE and WORD never have an operational sign!
LONG variables can have positive or negative values.

This leads to the following value range:

Thus, with inquiries such as:

IF A >= 0 THEN ...

Please ensure that “A” is of the type LONG. As a BYTE or WORD this IF inquiry would be rather meaningless since it would be true in any case always.

A negative integer is always generated as LONG, thus e.g.:

-33 = FFFF FFDDH

If this constant is now assigned a byte variable “B”:

B = -33

B contains the value: DDH = 201 (dec). The upper 24 bits will be simply cut off. The same applies for WORDs. Please always pay attention to the operational signs.

 

An example:

 

BYTE B1, B2
LONG L1, L2

L1 = -6
B1 = 30
B2 = L1 ' assign signed LONG
' to BYTE (unsigned)
IF B1 > B2 THEN ' here unexpected result
    ...

 

In this case B2 is assigned the value of L1 (=-6). The value -6 corresponds to =FFFF FFFA in hexadecimal notation. When assigning this LONG value to byte variable B2 the upper 3 bytes are clipped, producing the value B2 = FA (hex) = 250 (dec.). The comparison: IF B1 > B2 THEN .... thus produces the possible unexpected result that B1 is not greater than B2.

The same applies to arithmetic operations:

BYTE B
LONG L
B=0 						' B = 00H
B=B-1 						' B now gets the values FFH = 255
L=7FFF FFFFH 				' L <- greatest positive number
PRINT #2,L 					' print L as decimal: 2147483647
PRINT #2,L+1 				' L+1 = -2147483648 = smallest number

 

The ends of extent of integer types can be easily exceeded by arithmetic operations without this leading to a formal error. On the other hand, the programmer himself must check whether he wants such a result.

When calculating with whole numbers one normally uses LONG types which display the biggest range values and have an operational sign. Arithmetic operations with LONG will be executed much faster than REAL calculations and whole number are also suitable for comparisons such as:

IF A=B ..., or IF X <> 87654 THEN ...

Or as control variables or abort conditions in loops.

When using REAL remember that comparisons of the type:

IF R1 = 12.345678 THEN ...

Should never be used since in all probability they will never be TRUE or with “<>” never FALSE.

LONG variables can in fact be very useful for fast running fixed point calculations. These use a re-scaling of numerical values, e.g. by calculating “m°C = milli-degrees centigrade instead of °C. The number 123456 then means: 123.456 °C.

Remember to observe the LONG range of values with all intermediate results. Value overflows lead to inapplicable results:

Still OK:

46 340 * 46 341 = 7FFF 5D14H = 2 147 441 940

Overflow:

46 340 * 46 342 = 8000 1218H = -2 147 479 016 instead of 2 147 488 280

Example: If you would like to multiply a LONG-value “N” whose range of values is defined as -100 000...+100 000 by a fixed point number “1,234”, one proceeds in two steps:

  • Multiplication with 1234 - LONG number range is not exceeded.
  • Division by 1000 - also OK

Thus: RESULT = N * 1234 / 1000 ‘ first “*”, then “/”

Although LONG variables are signed, they can also always be regarded as a group of 32 bits. This means that all 32 bits, including the sign bit, can be manipulated with logic operators (BITAND, BITOR, BITXOR, ..), slice operators (SHL, SHR, ...), and functions (BITNOT), mirrors (BIT_MIRR), etc. The sign only becomes important when we see a LONG variable as a numerical value, in particular as a decimal number (see: PRINT and PRINT_USING). The sign does not exist as a bit pattern

 

For example:

L1=7FFF FFFFh 					' the greatest positive number
PRINT #LCD, L1
PRINT #LCD, L1+1 				' overflow to -0
PRINT #LCD, L1+2 				' the greatest negative number
USING "UH<8><8> kV8.8.8.8.8" 	' output the same in HEX
PRINT USING #LCD, L1
PRINT USING #LCD, L1+1
PRINT USING #LCD, L1+2
L1=8000H
PRINT #LCD, L1 SHR 1
PRINT #LCD, L1 SHR 5
PRINT #LCD, L1 SHR 8
PRINT #LCD, L1 SHR 17 			' result becomes 0, since the bit
' disappears
PRINT #LCD, L1 SHL 16 			' result becomes -0

An advanced course in Tiger MultitaskingFormat conversions

Format conversions

It is also very useful to have a good idea of the data formats in Tiger Multitasking and their conversions. Even just to generate shorter and faster running programs or to avoid programming errors from the start.

When converting data types we consider two areas:

  • Automatic conversions carried out by the compiler
  • Explicit conversions used specifically by the programmer

Numerical variables are converted automatically.

The variables of the type BYTE, WORD and LONG form a group whose members can almost always be used at random anyway. The compiler will initially convert all BYTEs or WORDs to a LONG format before every arithmetic operation, then carry out the calculation and finally create the result type. The programmer alone controls whether this leads to intentional or unintentional number value overflows.

You should also be aware that with I/O- instructions, the number of bytes corresponding to the type will be transferred:

 

BYTE1 = 77H
WORD1 = 66H
LONG7 = 55H
PUT #23, BYTE1,WORD1,LONG7

 

Transfers these data bytes to the I/O channel #23:

„77 66 00 55 00 00 00“

With a mixed use of REAL and integer types (BYTE, WORD, LONG) the compiler initially converts all integer types to REAL, calculates the expression and then produces the desired result type, e.g.:

PRINT #2, 4 * SIN (4)

Prints a REAL result: -3,027209981231716 E+000

Accordingly:

  • 1.234 * 8 ‘ REAL
  • 5/4 ‘ LONG
  • 5./4 ‘ REAL

STRING, ARRAY or FIFO are not automatically converted.

You should preferably program in the way that the compiler has to carry out no or only few automatic type conversions from integer to real and vice versa. Type conversions cost both computing time and memory space and possibly reduce the accuracy of the calculation. Avoid unnecessary type conversions. Example:

N=123.45*SIN(X/10)

If N and X are integer variables 2 type conversions will be executed here.

With explicit type conversions the intentions are the result of the requirements on the project. This can be e.g. the analysis or synthesis of a data stream, the conversion into a different display form or simply the improvement of the program flow.

The function for the type conversion transfer numerical values: BYTE, WORD, LONG, REAL to STRINGS or fetch them from STRINGS.

Since one always has free access to every arbitrary string position, strings can be compiled to arbitrary structures containing all sorts of different data types: Text, graphic, numerical value, ... etc. Such structures will be used again and again, e.g. in data bases and RDT applications.

An advanced course in Tiger MultitaskingSpeed

Speed

Some considerations for Tiger Multitasking programs execution speed:

In applications that have a high data throughput, include complicated calculations, or have to provide fast response times, it is worth while paying attention to factors that will increase the execution speed during programming.

 

Measuring the speed

The execution speed of Tiger Multitasking instructions and functions can be easily measured. The sub-directory ‘APPLICAT’ contains the example program ‘SPEED1.TIG’ to check the runtime behavior of instructions. This program can also be used to investigate speed aspects of one’s own program constructions and to compare these with alternative programming techniques. Shown below is an extract of the program example SPEED1.TIG:

 

TIME2=TICKS() 									' <- get Time now
LOOP 10000 										' 10.000 Schleifendurchlaeufe
    LL_IPORT_PULSE 8,0FFH,200 					' sende 200 Pulse
ENDLOOP '
TIME2=DIFF_TICKS(TIME2)-TIME10 					' <- 2M Pulse
PRINT USING #1,"Pls1:";2000000000/TIME2;"/s' 	' <- Pls1/Sek zeigen
WAIT DURATION 2000 								' 2 Sek warten

 

PC-Mode

Multitasking instructions always run slower in the PC mode than in the RUN mode. Moreover, the performance characteristics of a program, particularly multitasking applications, are completely different in the PC mode than in the RUN mode. Some instructions become very slow; while other instructions change only a little, if at all. These include instructions such as WAIT DURATION, or buffered I/Os, when their time behavior essentially depends on the current data traffic. Note that those instructions affected by mode change, are all ‘fast’ instructions.

This mode change can lead to surprising effects. For example, a task may run faster in the PC mode than in the RUN mode, and relative task performance is very different in the PC mode than it is in the RUN mode. If you wish to study the performance characteristics of an application, this should always be carried out in the RUN mode.

 

Numeric system

Tiger Multitasking uses fast, 32-bit integer arithmetic and the highly accurate double precision REAL arithmetic. If you are interested in speed, you should plan your application so that only integer arithmetic (BYTE, WORD, LONG) is used in the corresponding program phases wherever possible, and REAL arithmetic is only used when high speed is not required.

The 32-bit integer arithmetic is normally accurate enough for a number of applications. As an example, a data range of -20,000 to +20,000 can be generated with a resolution of 0.00001. This range provides 2,147,483,648 negative values and 2,147,483,647 positive values. Each of the values within this data range could represent measured values such as temperature, pressure, travel, speed and many more.

However, if function values are to use REAL type variables during program execution, system performance may be reduced. To counter this, a large table containing integer values (an ARRAY or STRING) can be formed at the start of the program. The integer values contained within this table can be used for value calculations, whereby providing faster calculation times.

If such tables are scaled with a factor, that is a power of 2, i.e. 2, 4, 8, 16, the slightly slower REAL type variable divisions can be replaced by the faster integer operations SHR (see Operators, from page 20). Example: Sine table with a scaling factor of 16384.

Calculations are always performed faster if Multitasking instructions contain the largest possible number of operations in one line. As an example, 5 additions in 1 line run approximately 3 to 4 times faster than 5 lines with only one addition each.

 

Interim results

Interim results which are used a number of times are managed as variables and do not have to be repeatedly calculated. The possible number of variables is practically unlimited if one considers that a LONG value occupies only 4 bytes and a REAL value only 8 bytes. Even the smallest Tiger module with 128 KBytes of RAM can accommodate may thousands of such variables.

As already mentioned above, unnecessary type conversions should be avoided. They reduce the accuracy, take up CPU time and enlarge the program code. This is particularly true of automatic type conversions between REAL and LONG. Changes between types BYTE, WORD and LONG are not affected by this. These can be mixed in an expression without any noticeable drop in speed of execution.

 

Strings

There are a number of new string functions for fast and effective string handling.

Repetitive tasks are implemented as functions that run very quickly and only require 1 or 2 lines of Multitasking code each. A further important factor when handling strings is that it takes a long time to copy strings, particularly if these are very long.

Instead of:

A$ = LEFT$ (A$,40) + „XYZ“ + RIGHT$ (A$,(LEN (A$)-3-40))

It’s much better to write:

A$ = STOS$ (A$, 40, „XYZ“, 3)

STOS$ (String-To-String) copies part of a string directly into a specified point of another string. This is much faster than the first example, which triggers a number of time-consuming copying processes. Further functions of this type are e.g. NTOS$ (Num-To-String), RTOS$ (Real-To-String), NFROMS (Num-From-String), RFROMS (Real-From-String). Another example:

Instead of:

N = ASC(MID$(A$,POS,1)) + (256*ASC(MID$(A$,POS+1,1)))

It’s much better to write:

N = NFROMS ( A$, POS, 2)

The string functions SELECT$ and INDEX are beneficial as they keep the program code understandable and to reduce execution time. SELECT$ and INDEX are also used to program menu selections, create data compression, command decoders and complex code converters, implement fast program branches, etc.

 

Example:

SEL=2
CHOICE$ = &
":left door :upper door :Ventilator-1:Ventilator-2:Emergency ::"
PRINT #LCD, Z2$; SELECT$ ( CHOICE$, SEL)

 

SELECT$ contains a series of entries (records), separated by a break character (in this case “:”). The function SELECT$ then selects the desired entry via the index SEL, in this case: “Ventilator-1”. This can be used to provide operating personnel with scrolled selection fields on a display, or frequently used expressions or terms can be abbreviated by an index, which can lead to enormous savings in space.

The reverse function here is INDEX. Let’s assume that you have a main string and a string with a keyword or byte sequence and you want to look for the index of the key in the main string.

Example:

 

SEL=2
COMMAND$ = "RIGHT"
COMSET$ = "-START-STOP-HALT-LEFT-RIGHT-UP-DOWN-RETURN-EXIT--"
N = INDEX (COMSET$, COMMAND$)

 

Index provides the value 4 into the variable N. This is the index of the term “RIGHT” in the string COMSET$. This enables the creation of a command interpreter with only a few lines of program. Another example follows:

Example:

COMMAND$ = "Return"
COMSET$ = "-START-STOP-HALT-LEFT-RIGHT-UP-DOWN-RETURN-EXIT--"
N = INDEX (COMSET$, UPPER$ (COMMAND$) )

 

Program loops

Program loops and particularly nested loops, can quickly become very demanding of processor resources. Just consider 3 nested loops of 30 runs each, this equates to; > 30 * 30 * 30 = 27000 loops!

An alternative structure with fewer runs could provide many performance advantages.

In multi-tasking environments, one must always remember that other tasks use free CPU time if one task no longer has anything to do. This is particularly true of test loops. Irrespective of whether the system is waiting for a key entry, limit switch, serial transfer or the like: if the corresponding task has been tested and the desired condition has not yet occurred, the remaining CPU time for this task should always be released for other tasks. This is carried out with the instruction RELEASE_TASK. The same effect with an additional dead time is achieved with WAIT_DURATION <ms>. This also immediately releases CPU time and this task receives no further CPU time for a period of <ms>.

Example - wait for key input:

 

SEL=2
CHOICE$ = &
":left door :upper door :Ventilator-1:Ventilator-2:Emergency ::"
PRINT #LCD, Z2$; SELECT$ ( CHOICE$, SEL)

 

In this example, a large part of the overall CPU performance is spent in a senseless WHILE-loop. If no key input has been immediately made by the user, then this does not have to be checked 0.1millisecond later. One could sensibly wait for 10, 20, 30 milliseconds or more. See the next example:

A$=""
FOR N = 0 TO 0 STEP 0
    GET #1, 1, A$ ' wait for key entry
    WAIT_DURATION 50 ' release CPU-time
NEXT
' process entry

 

IN, OUT

The instructions IN and OUT are available to set or read the digital pins of the Tiger Multitasking. These instructions always test whether the desired operation would interfere with the work of a device driver. This has the advantage that one does not accidentally trigger a reaction in a device connected via a device driver by addressing an I/O pin, which is already used by another device driver. Of course, in terms of speed this does cost runtime. This means that if you plan what pins do what correctly from the very beginning, the faster low level instructions such as LL_IPORT_IN and LL_IPORT_OUT can be used in such situations, which run correspondingly faster.

 

Pulse

There is also a quick and easy way of setting a certain number of pulses at a pin with the instruction LL_IPORT_PULSE instead of using a program loop with OUT instructions. As an example, LL_IPORT_PULSE can be used to set a 1 or 2-position digit display with a value or to start a different circuit with counter and function selection

 

Functions

The constantly growing library of functions forms a powerful tool for the effective implementation of programming tasks with few instructions. Tiger Multitasking functions cover the areas

  • Integer arithmetic
  • Floating-point arithmetic
  • String operations
  • Special functions.

It is possible that a repetitive programming task can be replaced by an individual complete function to offer high processing speeds and a compact code.

Multitasking

A number of tasks can access the same device driver. This should be used with care to retain the plausibility of any results. In the following example, the two tasks ‘Count1’ and ‘Count2’ display their counter level on the LC-display. The numbers are output mixed. The flag, ’F’, is switched over with a keystroke so that one task now “outputs” the counter level in an ordered manner while the counters continue to run:Program example:
 

'---------------------------------------------------------------------
' Name: LCD-OUT.TIG
'---------------------------------------------------------------------
LONG C1,C2,N,F          			   		' global vars of type LONG
TASK MAIN               	  				' begin task MAIN
    STRING A$                 				' var of type STRING
    ' install LCD-driver (BASIC-Tiger)
    INSTALL DEVICE #1, "LCD1.TDD"
    ' install LCD-driver (TINY-Tiger)
    ' INSTALL DEVICE #1, "LCD1.TDD", 0, 0, 0, 0, 0, 0, 80h, 8
    RUN_TASK Count1,50             			' start Count1 w. prio. 100
    RUN_TASK Count2,10             			' start Count2 w. prio. 50
    F = 0                 					' reset flag
    LOOP 9999999                		 	' many loops
        FOR N = 0 TO 0 STEP 0         		' endless loop until N=1(GET!)
            RELEASE_TASK             		' release rest of task time
            GET #1, #0, #1, 1, N      		' N=chars in keyboard buffer
        NEXT                 				' end of endless loop
        GET #1, 1, A$            			' read char f. keyboard buffer
        F = (F BITXOR 1)           			' toggle flag
        IF F = 1 THEN          				' if flag=1:
            RUN_TASK CPrint           		' start task CPrint
        ELSE                 				' else:
            EXIT_TASK CPrint      			' exit task CPrint
        ENDIF                 				'
    ENDLOOP                 				'
END                     				    ' end task MAIN
'---------------------------------------------------------------------
' Task Count1: Counts up C1 and prints current value to LC-display
'---------------------------------------------------------------------
TASK Count1                	 			    ' begin task Count1
    FOR C1=0 TO 9999999             	    ' FOR-loop counts up C1
        IF F=0 THEN                		    ' if flag=0:
            PRINT #1, C1        		    ' output of C1
        ENDIF        					    '
    NEXT                   				    ' next value for C1
END                     				    ' end task Count1
'---------------------------------------------------------------------
' Task Count2: Counts up C2 and prints current value to LC-display
'---------------------------------------------------------------------
TASK Count2                 				' begin task Count2
    FOR C2=0 TO 9999999             		' FOR-loop counts up C2
        IF F=0 THEN                 		' if flag=0:
            PRINT #1, C2             		' output of C2
        ENDIF                 				'
    NEXT                     				' next value for C2
END                     					' end task Count2
'---------------------------------------------------------------------
' Task CPrint: Output of counters C1 & C2 to LC-display
'---------------------------------------------------------------------
TASK CPrint                 				' begin task CPrint
    LOOP 9999999                 			' many loops
        PRINT #1, "<1>"; C1             	' output of C1
        PRINT #1, C2             			' output of C2
        WAIT DURATION 100             		' wait 100 ms
    ENDLOOP                	 				'
END                     					' end of task CPrint

Such effects can be seen in the following example. A device drivers’ output channel uses a buffer. Two tasks, X and Y, use this buffered channel to access an output device. However, task X repeatedly succeeds in accessing the device to convey its output data, but task Y finds that the buffer is continually full. Consequently, task X is operative while task Y is not. The programmer may get the impression that task Y is at fault, when in reality it is the buffer management that is at fault. Note that this effect may change so that task Y has use of the channel while task X has not. Whichever way the effect happens, the operator sees an inexplicable discontinuity.

Remedy:

Either one avoids a situation where 2 tasks use one and the same device, or a corresponding manager is installed (subroutine or task) which can sensibly coordinate the requirements of various tasks and the characteristics of the device.

WAIT instructions, just like I/O instructions with buffered device drivers, modify the time behavior of the program sequence. A WAIT instruction, when waiting, immediately releases CPU time for other tasks. The same thing happens when a device driver buffer is empty/full (Input / Print). This shifts the relative performance of all tasks. Tasks which receive WAIT instructions or I/O instructions on buffered channels always release CPU time. Thus, in the long term they tend to take up less CPU time than is due to them on the basis of their priority.

If a number of tasks are working jointly with common global variables, care must be taken to ensure that an invalid condition never arises. Such conditions can occur when a number of tasks are executed (quasi) simultaneously.

Example:

A task wants to execute a manipulation of an array and checks the corresponding semaphore to see whether the array is currently free or blocked. If the semaphore signals “free”, the task itself sets the semaphore to “blocked” and starts its work. The critical phase in this case is between the two points in time ‘Check semaphore’ and ‘Set semaphore to “blocked”’. Since we are in a multitasking environment, there is always the risk that task switching will occur immediately after the semaphore is checked and a different task will carry out the same sequence (check semaphore and set semaphore to “blocked”). If the first task executes its instruction a little while later, “blocking” the semaphore, the semaphore has already been set and the second task also believes that it can manipulate this array. Data corruption follows.

This conflict is avoided by stopping the switch task mechanism in the critical phase between checking and setting the semaphore (DISABLE_TSW), and releasing this immediately after the critical situation (ENABLE_TSW).

Initialization of variables

Strings, as well as Integer and Real variables, are not initialized by the type declaration. In the case of strings this means that both the length and content of the string are completely unknown after declaration, numerical variables bear an unknown random value.

In systems without a battery backed RAM, care should be taken to ensure that all variables are assigned values before being further evaluated. In systems with a battery backed RAM a differentiation is made between two cases:

  • Recovery after power-down, values are retained and are not initialized
  • First start or data loss in RAM: values are initialized.

Using a Watchdog

A watchdog is a hardware component, which will reset the unit after a certain amount of time, when it is not being served frequently. The Tiger module will be reset and eventually also other components on the pcb.

The benefit of this hardware watchdog is to reboot a stuck or faulty program.

Program errors, electromagnetic interferences or errors in memory can be noticed and avoided.

The operating mode of a watchdog is built on a charging and discharging of a capacitor. The charging time is 125 μs. The discharging time is about 20-30 s. After this time, a hardware reset will be performed.

Before the complete discharge, the watchdog has so be served, e.g. every second by inverting the watchdog-pin-level.

The port and pin number can be determined through the corresponding data sheet.

The watchdog is only active in RUN-mode. In PC-mode, the watchdog is inactive.

It can be served via hardware by setting the corresponding jumper. By software, the corresponding pin needs to be inverted.

 

Examples:

'' init watchdog pin

out 1, 00010000b, 0

dir_pin 1, 4, 0

'' calm down watchdog by inverting its pin level

out 1, 00010000b, 0

wait_duration 1000

out 1, 00010000b, 1

wait_duration 1000

'' ...

'' deactivate watchdog by low pc mode pin

out 4, 00000010b, 0

dir_pin 4, 1, 0
'' activate watchdog by setting pc mode pin as input

dir_pin 4, 1, 1

 

For easy handling in multitasking programs the modul TL_WATCHDOG of the Tiger Multitaskingt Library can be used.

All Tiger Multitasking subroutines, that take more than 1s will automatically serve the watchdog. More specifications can be found in the Tiger Multitasking Library manual.

Please consider:

Serving the watchdog in it’s own special task will only show few errors. It is highly recommended to serve the watchdog in endless loops.

The watchdog needs to be served separately, when subroutines will last more than 1 second.

It is also recommended not to serve the watchdog too often to keep the CPU load as low as possible.

The watchdog needs to be deactivated at the very beginning of a function, which will consume more than 1 second.

Here some functions where this is necessary:

 

update_me
erase_flash wait_duration
wait_next
wait_clock

 

When using wait_duration alternatively a loop with ticks/diff_ticks can be used which will serve the watchdog.

A more elegant way is the use of the Tiger Multitaskind Library function vTlWaitDuration when using the module TL_WATCHDOG.
