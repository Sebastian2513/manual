Graphic functions overview

A picture says more than a thousand words.

Graphic functions help applications make clearer and more understandable outputs. Pictures and graphics, drawings, diagrams, instruments, curves and much more can be shown on screen and printed out.

 

Some examples:

 

Graphics are easy to program in Tiger Multitasking and are characterized by:

  • high speed / fast picture generation,
  • finished functions for repetitive applications,
  • great flexibility through operations such as:
  • overlays
  • masking
  • shifts
  • scaling

All graphic functions run completely automatically in Tiger Multitasking so that no graphic adapters or special controllers are needed to generate and edit graphics. Graphic data is normally compiled in strings in the FLASH area of the Tiger.

The type of graphic output device is not important during the generation and editing of graphics. The needs of the respective graphic output device are only taken into account during the output of images and moving sequences by selecting the suitable device driver or through an independent Multitasking

routine.

A special graphic toolkit is provided as an aid for the developer which has the necessary hardware such as graphic screen, touch-screen, mouse input, etc. as well as numerous example applications. The example programs listed in the manual come from this toolkit and can be executed there immediately.

Tiger Multitasking essentially differentiates between 2 types of graphic display:

  • Pixel graphic
  • Vector graphic

Pixel graphic - Vector graphic

A pixel graphic is the 1:1 representation of pixels by individual bits. A ‘1‘-bit corresponds to a “black” dot or the foreground color, a ‘0‘ bit a “white” dot or the background color.

  • Bit value = “0”  white pixel
  • Bit value = “1”  black pixel

The 8 bits of a byte form a pixel group of 8 horizontally arranged pixels whereby the highest-value bit is the pixel on the far left:

 

 

The successive bytes in a string form the resulting pixel groups in the pixel data area:

 

 

And finally, a 2-dimensional display is obtained from a long “thread” of sequentially arranged pixels by starting at the beginning of a line again when a picture width of nn pixels is reached.

Example: 16 x 14 pixel graphic shown as pixel display and string (hex)

 

 

The same as a string (hex): „01 00 03 80 07 C0 0F E0 1F F0 10 10 12 50 ...“%

The Format width in the example above is 16 pixels, the height 14 pixels. From the type of pixel representation in the bytes of a string it is clear that the format width of a graphic is always a multiple of the number 8 since an even number of bytes can only ever be used per pixel line.

However, this does not restrict its general use if any other value which is not divisible by 8 is allowed for the Picture width to manipulate, move and generate graphics:

Here the example of a small graphic LCD display with 20 x 20 pixels (FORMAT

data!):

 

Although the graphic string for a format-filling image on this display contains 60 bytes (3 x 20) or 480 bits, only 20 x 20 = 400 pixels are shown here. The excess bits, shown here as “”, occupy memory space but are never shown.

The slight “waste” of bits in such cases is however, offset by the great advantage of a uniform bit-to-pixel assignment. A pixel line always begins with bit-7 of a byte.

Rules in Tiger Multitasking pixel graphics:

  • 1 Bit ↔ 1 Pixel
  • Bit value “0” = “white”, bit value “1” = “black”
  • 1 Byte = 8 bits = 8 horizontal pixels
  • Bit-7 → Pixel on the far left
  • The 1st byte of a string is in the top left hand corner of a graphic
  • The last byte of a (format-filling) string is in the bottom right of the graphic.

Significance of pixel graphics

Pixel graphics are the most common form of graphic display to output graphics on graphic screens (LC-displays) and printers (matrix, thermo, laser printers). Depending on the device type, the pixel representation may have to be reformatted for output on the output device.

Reformatting takes place either:

  • in the device driver,
  • through special graphic functions, or
  • through own BASIC routines

Applications where the source format of graphics is typically in a pixel representation include:

  • photos, low-resolution video
  • scanned line originals
  • company and special symbols
  • letters and numbers in a special design
  • very small pictures and symbols
  • moving animations, particularly of small dimensions
  • town plans ... etc.

Pixel graphics always occupy the same memory space irrespective of whether and what is shown in the pixel graphic. The memory requirements of pixel graphics with special structure features can be reduced by data compression. The memory space and compute time for compression and decompression should be given careful consideration.

One possible way of saving memory space through compression while still achieving short compute times could be to save packed pixel graphics in the FLASH data area and to unpack this once into variables (RAM) when the program is started. During the further program run the finished (unpacked) pixel graphics could then be accessed directly.

Editing possibilities with pixel graphics

Pixel graphics can be edited with the following instruments in the Tiger Multitasking:

  • copy
  • cut
  • mirror
  • invert
  • overlay
  • shift
  • mask

Editing a pixel graphic requires little CPU-time in the Tiger Multitasking, so that pixel graphics can be edited quickly and are ideal for moving animations.

 

Coordinate system for pixel graphics

The Cartesian coordinate system is used for pixel graphics with a downward pointing Y-axis, corresponding to the features of typical output devices such as graphic LCD or matrix printers (compare: vector graphics).

 

 

You can work in all 4 quadrants with pixel graphics, whereby only those parts of a pixel graphic which are in the “active graphic area” will be visible. If a graphic manipulation is carried out in “Window-1” this does not affect the actual pixel graphic in the “active graphic area”. A manipulation in “Window-2” on the other hand would affect the black marked overlap area and change the pixel graphic at this point.

This feature is exploited for example to create random sections from pixel graphics and move these freely on a screen. Such sections can cover the entire screen or only a small window in the “active graphic area”.

Vector graphics

In the case of vector graphics the picture level is covered by a coordinate network so that every point on the level can be assigned a coordinate value pair.Pixel assignment → Cartesian coordinate system:

 

 

Coordinate of the pixel = (11,3)

 

A special feature of this coordinate system is the downwards pointing Y axis. This leads to an inverse orientation compared to the commonly used layout with an upwards pointing Y axis. The reason for this orientation is the common assignment of memory to pixels in graphic LCD’s and matrix printers as well as the temporary assignment of pixels when certain pixel lines are to be printed out and transferred.

The Cartesian coordinate system ideally reflects the relationships in the majority of graphic I/O devices:

  • monitors
  • printers
  • laser systems
  • NC machines
  • scanners ... etc.

The Cartesian coordinate system in the Tiger Multitasking works with the following value ranges and resolutions:

X-/Y-coordinates: -8191...0...8191, Step = 1

 

 

The area marked in the picture as “active graphic area” corresponds to the area covered by a string with pixel data. This can also be the visible area of a graphic LC display, for example. With vector graphics both negative and positive coordinate values are allowed, though vector graphics always limit instructions to the active graphic area. If you draw beyond the margins of the active graphic area the line is automatically limited to the margin.

Polar coordinate systems are also in widespread use, particularly where physical properties favor these systems, e.g.:

  • radar systems
  • ultrasonic diagnosis
  • directional characteristics of radiators / sensors, etc.

Tiger Multitasking also supports polar coordinates which offer simplifications in corresponding applications. However, one should always pay attention to the iteration errors which occur when changing from one coordinate system to the other.

The polar coordinate system in the Tiger Multitasking works with the following value ranges and resolutions:

Radius (R): 0...8191, Step = 1

Angle (PHI): -36000...36000, Step = 1 = 0.01 degrees

Orientation of the polar coordinate system:

Vector graphics enable very flexible visual realizations. Vector graphics images are described by a sequence of coordinate pairs and additional parameters. Coordinate pairs can come from different sources, e.g.

  • from tabular data,
  • from calculations,
  • from measurements.

Since coordinate pairs can be easily subjected to mathematical mapping, illustrations with vector graphics can be subject to a number of transformations and manipulations.

Some transformations are already supported in the functions of Tiger Multitasking for easier handling and a faster runtime behavior:

  • rotation
  • scaling
  • shift
  • coordinate transformation
  • area filling

If they are not too complex, vector graphics allow the storage of illustrations in little memory space.

Working with pixel graphics

Pixel graphics is understood as being the representation of "black" and "white" dots (pixels) by an assigned bit value of "1" or "0". This pixel data are normally saved in Tiger Multitasking in strings or the Flash data area.

Pixel graphics can be generated in various ways, for example the vector graphics functions of Tiger Multitasking, reading in graphics via interfaces or own Multitasking programs. A further possibility is to import graphics from other programs (graphic programs, CAD, DTP, etc.). The "Bitmap" format (*.BMP) can be imported directly into a Tiger Multitasking program.

 

Graphics in Flash

The GRAPHFLT filter is used to transfer this type of data into the internal format of the Tiger:

DATA FILTER "JDOL.BMP", "GRAPHFLT", 0 ' mozart

This instruction loads the file "JDOL.BMP", containing a bitmap graphic, to the data area of the FLASH and hereby converts this into the described, internal pixel graphic format.

 

Some tips on generating bitmap graphics which can be used in Tiger:

  • Always select a format width in integer multiples of 8. This corresponds to the con ventions in Tiger Multitasking and means no limitation to the general features for further processing of these graphics.
  • You can select a random height for the graphic.
  • Select the type "black/white graphic". This means that a bitmap graphic with only two color values is generated: black and white pixels (not: grey-scale or color graphics).
  • If you wish to use pictures with color or grey-scale values this can be brought into a displayable form by matrices. There are various types of matrix and methods. Dot matrix and dithering are common.
  • If you wish to design animation sequences you may have to take into account the optical sluggishness of LC-Displays. This means that fast changes and fine lines are not displayed as expected.

Graphics and speed

Using the graphic functions presented here for vector and pixel graphics the developer has at their disposal a wide range of possibilities for optically appealing and informative outputs on displays and printers. No special graphic boards or controllers are needed for graphic processing and output. All operations for picture generation and processing as well as output on a display run in Tiger Multitasking.

This should be taken into account when planning applications where both:

  • the very intensive use of graphics as well as
  • very fast output sequences (films, animations ...)

are required.

The already very fast graphics in Tigers Multitasking can be used even better and additional CPU time won for other tasks by good planning and with a few tips.

Here are a few tips if you are developing graphic-intensive applications:

 

 

 

'---------------------------------------------------------------------
' Name: GRAPHIC_PEEK.TIG - PIXEL graphic: import, display pixelgraphic
' initializes the LCD and ouputs a graphic peeked out of flash memory
'---------------------------------------------------------------------
' Notes: set correct path for flash data (OPTIONS)
' connect L71 to reset LCD
'---------------------------------------------------------------------
USER_VAR_STRICT
#include DEFINE_A.INC                                   ' general definitions
#include UFUNC3.INC                                     ' definitions for user function codes
#include LCD_4.INC                                      ' definitions for LCD Typ 4
#include GR_TK1.INC                                     ' definitions for Graphic Toolkit
STRING SCREEN$(4K)                                      ' Global Strings for Pixel-data
DATALABEL PIX_DATA
TASK MAIN
    PIX_DATA::
    DATA FILTER "MOZART1.BMP", "GRAPHFLT", 0
    DIR_PORT 8,1000000b                                 ' L80...L86 output: LCD ctrl
    DIR_PORT 7,11111101b                                ' L71 output: LCD reset
    OUT P_LCD_RESET,M_LCD_RESET,0                       ' RESET LCD
    OUT P_LCD_RESET,M_LCD_RESET,255                     ' "
    OUT P_LCD_CTRL,M_INVERT,255                         ' CTRL pin of T6963C: no invert
    OUT P_LCD_CTRL,M_FONT_SEL,0                         ' CTRL pin of T6963C: font
    OUT P_LCD_CTRL,M_DISP_ON,255                        ' CTRL pin of T6963C: display ON
    
    INSTALL_DEVICE #12,"LCD-6963.TDD",0,0,0EEH,4,150,11H' 240x128,150KB/s
    PEEK_FLASH PIX_DATA, SCREEN$, 3820
    PUT #12, #1, SCREEN$, 0,0,3840                      ' <-- print graphic to LCD
END

 

Graphics with PEEK_FLASH

The following example shows the import and screen output of the graphic "JDOL.BMP" on a 240 x 128-pixel graphic display (T6963C). Data from the PC's files system can be loaded in the Tiger Flash data area with the instruction DATA and supplement "FILE" or "FILTER" in Tiger Multitasking programs.

Specification of the filter name "GRAHFLT" and the parameter: 0 means that the compiler converts the "*.BMP" data format to the internal pixel format of the Tiger.

The graphic is read with the instruction PEEK_FLASH. "SCREEN$" accepts the graphic and transfers this for printout to the device driver (PUT #12, ...).

Pay attention to the correct specification of the path name for the file to be imported when compiling this example, either together with the file name or generally applicable under "OPTIONS - DIRECTORIES" in the interface menu bar.

Graphics directly from Flash

This example generates the same printout as the previous example program. The only difference is that the instruction PEEK_FLASH is not used since pixel graphics can also be output directly from the flash data area. This simplifies the output of standard graphics, present as libraries in the flash, without them having to be read in as a RAM variable beforehand.

 

Program example:

'---------------------------------------------------------------------
' Name: GRAPHIC_DIRECT.TIG PIXEL graphic: import,display pixel graphic
' initializes the LCD and ouputs a graphic direct from flash memory
'---------------------------------------------------------------------
' Notes: set correct path for flash data (OPTIONS)
' connect L71 to reset LCD
'---------------------------------------------------------------------
USER_VAR_STRICT
#include DEFINE_A.INC 								    ' general definitions
#include UFUNC3.INC 								    ' definitions for user function codes
#include LCD_4.INC 									    ' definitions for LCD Typ 4
#include GR_TK1.INC 								    ' definitions for Graphic Toolkit
DATALABEL PIX_DATA
TASK MAIN
    PIX_DATA::
    DATA FILTER "MOZART1.BMP", "GRAPHFLT", 0
    DIR_PORT 8,1000000b 								' L80...L86 output: LCD ctrl
    DIR_PORT 7,11111101b 								' L71 output: LCD reset
    OUT P_LCD_RESET,M_LCD_RESET,0 						' RESET LCD
    OUT P_LCD_RESET,M_LCD_RESET,255 					' "
    OUT P_LCD_CTRL,M_INVERT,255 						' CTRL pin of T6963C: no invert
    OUT P_LCD_CTRL,M_FONT_SEL,0 						' CTRL pin of T6963C: font
    OUT P_LCD_CTRL,M_DISP_ON,255 						' CTRL pin of T6963C: display ON
    INSTALL_DEVICE #12,"LCD-6963.TDD",0,0,0EEH,4,150,11H' 240x128,150KB/s
    PUT #12, #1, PIX_DATA, 0,0,3840 					' d i r e c t l y to LCD
END

Bitmap too high

The following program example outputs a pixel graphic which has been exported in the graphic program with an incorrect height of 190 instead of 128 pixels. Consequently the picture appears "stretched” and only part of the picture can be seen. This graphic also takes up more byte memory space in the data area of the FLASH than necessary:

 

Screen size: 240 x 128 Pixel = 30720 Pixel = 3840 Bytes

Occupied FLASH-memory: 240 x 190 / 8 = 5700 Bytes

 

Program example:
